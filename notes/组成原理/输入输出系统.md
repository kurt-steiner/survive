# 输入/输出系统

![image](../images/输入输出系统.png)

在 **中央处理器** 这个章节中，我们用函数的视角去看待CPU的功能  
![image](../images/中央处理器.png)

我们先看 **输入** 部分  
我们对CPU输入了 指令，异常或者中断，其中

1. 指令来自于内存
2. 异常来自于CPU内部
3. 中断来自于 **外部设备** 或者 **DMA**

在传统计算机中，外部设备产生了一个字节的数据时，会通过中断通知 CPU，告知有数据产生，需要处理  
此时CPU会通过通过 I/O 指令，从设备的寄存器中读取值，并转移到内存中  
现在计算机中，DMA大行其道，他处在 内存 和 外部设备 之间，会将 外部设备 产生的数据搬运到 内存中，当数据准备完毕后，通过中断 通知 CPU 处理  
因此我们可以认为，在发起中断前，数据就已经在内存中准备好了

我们再看 **输出** 部分

1. CPU会将运算结果存储到 主存中
2. 或是持久化存储到硬盘中
3. 或是将自己的文件通过网络传输给另一个设备
4. 或是将图形数据发送给显卡，渲染到显示器上

其中 CPU 不需要直接访问数据，直接告诉DMA把内存中 从哪到哪 的数据写入设备即可  
当 DMA 完成任务后，通过中断 告知CPU，已经完成了任务

可以看出，我们不仅要支持 **对主存** 进行读写(其实CPU已经支持)，也要支持 **对外部设备** 进行 读写  
也就是说，外部设备可以作为数据的生成者，也可以作为数据的消费者

至此，我们的任务已经非常明确，我们需要探讨

1. 外部设备有哪些
2. 如何访问外部设备
3. 如何处理数据，包括 读/写 ，发送/接收

## 外部设备

外部设备可以用作输入，也可以用作输出  
说到输入，我们都会想到 键盘，鼠标这类东西，其实硬盘这个东西也可以用作输入设备，比如我想从硬盘中 **拿出** 一个文件，传给另一个设备

输出的情况也有很多，我们提到的硬盘也是一个，可以把小电影存在里面  
还有一些设备，比如 显示器，网卡，打印机这些 也可以看作输出设备

## 如何访问外部设备

书上说，主机和外部设备之间的访问是通过IO接口，他的表达不够详细，也不够准确  
我们知道

1. 在主机和显示器上，各有一个HDMI接口，用一条HDMI线将主机内的数据渲染到显示器上
2. 在主板和硬盘上，各有一个 sata 接口，通过一条 sata 线支持 主机对硬盘的访问

可以看到，一个主机内不一定只有一个I/O接口，不同的功能之间需要不同的I/O接口

___

![image](../images/IO接口基本结构.png)
我们知道，CPU是通过系统总线 对其他设备进行交互的，其中有

1. 数据总线
2. 地址总线
3. 控制总线

如果把I/O接口看作一个函数，他处在 主机和外部设备之间  
他从主机侧接收到了

1. 来自数据总线 的 数据
2. 来自地址总线 的 地址
3. 来之控制总线 的 控制信号

他从外部设备侧收到了

1. 外部设备的状态
2. 外部设备的数据

经过I/O控制逻辑的处理后  
输出给主机

1. 数据，通过数据总线
2. 地址，通过地址总线
3. 控制，通过控制总线，可能是中断信号

输出给外部设备

1. 数据
2. 控制信号

___

将I/O接口的基本结构进行提取，抽象，变为透明，我们现在要通过 I/O 接口对外部设备进行读写

我们知道，在 unix/linux 中，一切皆文件，各种设备被抽象成 `/dev` 目录下的各种文件  
在进行设备操作时，我们可以

```c
int fd = open("/dev/fb0", O_RDWR);
write(fd, pixel_data, size);  
```

为什么这样设计呢，在 unix 诞生之初，大概是 1970 年，那个时候设备种类相对简单，显卡都还没诞生  
创始人 **Ken Thompson** 和 **Dennis Ritchie** 追求简洁统一，干脆简洁到底，规定了一切皆文件

在unix/linux系统下，我们可以从内存中访问到设备对象，对其进行操作  
换句话说，我们将设备对象**载入到了内存** ，这种方式叫作 **统一编址** ，其统一在

1. 我们可以用统一的地址空间去 访问到设备
2. 我们可以用统一的I/O指令去 读写设备

后来外部设备种类 变得繁多，unix 方式 应对如此多的外部设备，和兼容新老版本设备之间的差异，显得力不从心  
后起之秀，**巨硬**公司的 windows 系统采用了另一种方案 **独立编址** ，在系统中除了 主存外，还有一个单独的 "内存" 用来存储这些设备对象  
由于是独立的，我们需要

1. 指定不同的地址空间去 访问到设备
2. 使用不同的读写指令去 读写设备

## 如何处理数据

在早期计算机中，外部设备产生了1个字节的数据，就会发起中断信号给 CPU，CPU接收到中断信号后，会通过总线将数据送到内存中  
如果是键盘，鼠标这些 低速设备，这种 **中断方式** 完全够用  
如果是高速外设，比如SSD硬盘，每秒的传输速度是200Mb，再每1字节数据中断一次的话，会让CPU无暇顾及任务，直接中断到几百年后，可以看作是完全死机

我们在初中时就学过 大脑和小脑，其中小脑用来控制 呼吸，憋尿 这些底层功能，如果大脑每时每刻都用来控制呼吸，那我们基本就没法思考了

在计算机中，为了应对这种大量中断的情况，我们有**DMA**这个小脑来承接 高速外设 的数据传输工作，当数据传输完成后，DMA通过中断告诉CPU，任务已经完成

同时，我们需要注意到，大脑其实能够手动控制呼吸，CPU也可以直接进行IO操作  
并且，中断 和 DMA 的应对场景不同，两者之间不能完全替代  
接下来我们需要对 中断 和 DMA 进行详细探讨

### 硬件处理/中断

中断方式 用来应对少量数据，也用来处理一些高优先级的事件

在编程领域，有个术语叫作 **原子性** ，意为这个操作不能被中断，不然数据有可能错误  
而CPU实现这个原子性，需要屏蔽所有中断，这个操作通过 **关中断** 指令实现  
到了需要接收中断时，使用 **开中断** 指令实现

我们来看这个视频 [又快又急的屎](https://www.bilibili.com/video/BV1SA411F7c2?spm_id_from=333.788.player.switch&vd_source=4d1cda3d4adb59a9c01207e76b796f06)

视频中，原来在马桶上的那位，就是因为没有关门，外面的人进来直接一屁股坐在他身上开始放水 🤣  
这个门完全可以看作 开关中断，有关中断的原因，就是怕数据由于同步原因，出现错误
___
CPU正在运行一个任务，如果在指令中断周期，接收到一个中断信号 **Signal**，如果这个中断的优先级较低，那完全可以屏蔽掉  
如果CPU执行的是一个原子性的任务，即已经 关中断，也会屏蔽调这个中断  
也就是说，在接收中断前，得看 **CPU是否关中断** 和 **程序是否屏蔽**

假设处理中断时，没有其他中断产生  
接收中断后，就像 小脑的反射 和 大脑的处理  
这个中断先由硬件进行处理

1. 关中断
2. 保存断点/PC
3. 找到中断服务程序的 内存地址

再由软件进行处理

1. 保存现场
2. 运行中断服务程序
3. 恢复现场
4. 开中断，返回

这里注意，在最后，是先开中断，再返回/return  
如果是先return，在开中断的话，有种说法是，怕CPU处理过快，把开中断忽略掉了，这种解释我弄不明白  
我们暂且用上面的例子，把开中断比作打开厕所门，把返回比作做回到椅子上  
如果是先回到椅子上，再开厕所门，就有矛盾了 -- 想要回到椅子上，必须先开门

我们把视角切回汇编代码中，其实在调用函数时，都会进行保存现场  
对于如下代码

```c
int add(int left, int right) {
    return left + right;
}

int main() {
    int temp1 = 125;
    int temp2 = 80;

    int sum = add(temp1, temp2);
    return sum;
}
```

对其进行汇编后，我们看关键部分

```asm
add:
	pushq	%rbp
	movq	%rsp, %rbp
	movl	%edi, -4(%rbp)
	movl	%esi, -8(%rbp)
	movl	-4(%rbp), %edx
	movl	-8(%rbp), %eax
	addl	%edx, %eax
	popq	%rbp
	ret
	.size	add, .-add
	.globl	main
	.type	main, @function
main:
	pushq	%rbp
	movq	%rsp, %rbp
	subq	$16, %rsp
	movl	$125, -4(%rbp)
	movl	$80, -8(%rbp)
	movl	-8(%rbp), %edx
	movl	-4(%rbp), %eax
	movl	%edx, %esi
	movl	%eax, %edi
	call	add
	movl	%eax, -12(%rbp)
	movl	-12(%rbp), %eax
	leave
	ret
	.size	main, .-main
	.ident	"GCC: (Debian 14.2.0-19) 14.2.0"
	.section	.note.GNU-stack,"",@progbits
```

在 `main` 和 `add` 函数段中，起始都是 `push` ，也就是将当前现场/栈指针 压入内存中

如果有高优先级的中断信号传入，并且CPU允许多重中断，我们需要调整 软件处理 的阶段

1. 开中断 -- 表示允许嵌套
2. 运行当前 中断服务程序
3. 关中断 -- 表示嵌套结束

换一个说法来比喻这个过程 -- 中序遍历
