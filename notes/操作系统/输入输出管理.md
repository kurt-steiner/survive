## 设备独立性软件 -- 磁盘缓冲

不要看 **设备独立性软件** 这个名字这么唬人，我们知道操作系统是一种 介于 硬件与用户 之间的一种特殊软件，他对用户屏蔽了硬件细节，并最大化利用硬件性能  
同样的，设备独立性软件 对用户屏蔽了 输入输出设备硬件 的细节，并对 输入输出 事件进行优化

我们特别讨论下磁盘缓存  
我们知道输入输出 在整个操作系统中是最慢的一部分，在操作系统需要访问一部分磁盘内容时

![image](../images/disk-buffer-1.png)

```
read/write disk-blocks[start:end]
```

![image](../images/disk-buffer-2.png)

我们一共需要访问 $\text{count} = \text{end - start}$ 次磁盘，一共耗时

$$
\text{T(cost)} = \text{count} \times (T(传送到内存) + T(传送到内部) + T(处理数据块))
$$

每个磁盘块的访问时间是
$$
T = T(传送到内存) + T(传送到内部) + T(处理数据块)
$$

___
在引入缓冲区之前，你可能 已经注意到了 书上说的初始条件，你完全可以不用管他，他的意思，用人话来说，就是在下面这些图像中，处理的顺序需要按照

1. 从磁盘传送到内存
2. 从内存传送到CPU内部
3. CPU进行处理

这样的顺序来讨论一个数据块的处理时间
___
在CPU进行数据处理时，把磁盘中的数据载入内存 与 CPU 运算是完全不冲突的，我们完全可以提前读取一部分磁盘块到内存中  
我们在内存中创建 **一个缓冲区** 用来提前预读磁盘，在这种情况下，图像变成了

![image](../images/disk-buffer-3.png)

在 $T(传送到内存) > T(处理数据块)$ 时，每个块的处理时间是
$$
T = T(传送到内存) + T(传送到内部)
$$

在 $T(传送到内存) < T(处理数据块)$ 时，每个块的处理时间是
$$
T = T(处理数据块) + T(传送到内部)
$$

整理形式，每个块的处理时间为
$$
T = \text{max}(T(传送到内部), T(处理数据块)) + T(传送到内部)
$$

___

如果我们再加一个缓冲区呢？  
我们知道，两个缓冲区的数据传输是不冲突的，那数据的处理流程会是

![image](../images/disk-buffer-4.png)

在图像中，我们发现整体的时间 与 $T(传送到内存)$ 和 $T(传送到内部) + T(处理数据块)$ 有关

1. 当 $T(传送到内存) < T(传送到内部) + T(处理数据块)$ 时  
每个块的处理时间大致是 $T(传送到内部) + T(处理数据块)$
2. 当 $T(传送到内存) > T(传送到内部) + T(处理数据块)$ 时  
每个块的处理时间大致是 $T(传送到内存)$

## 磁盘使用

### 磁盘地址

![image](../images/disk.png)

描述磁盘中的地址，我们用

1. 柱面号
2. 盘区号
3. 扇区号

柱面号，或者说磁道号，指的就是  
![image](../images/disk-2.png)

这张跑道中的第几号跑道

由图中可知，一个磁盘由多个盘片构成，每个盘片都有正负两面  
假如有三个盘片，盘区号依次数下来是 $0,1,2,3,4,5$ 共6个盘区

扇区号，就是说将一个盘面分为多个扇区后的扇区在哪里

![image](../images/disk-3.png)

### 磁盘寻址

要寻址到相应的磁盘位置，先要寻址到 相应的磁道，再转到相应的扇区，最后进行传输

**寻址到磁道时**，需要花费
$$
T = \text{count(磁道数)} \times \text{speed(跨一个磁道)} + \text{time(磁臂启动)}
$$

**寻址到扇区时**  
![image](../images/disk-4.png)

如果磁头刚好要到达扇区的起始区，此时
$$
T = 0
$$

如果磁头刚好在扇区的末尾，此时
$$
T = \frac{1}{\text{speed(转速)}}
$$

平均一下，得到
$$
T = \frac{1}{2 \times \text{speed(转速)}}
$$

**最后是数据传输**  
我们需要读取 $\text{count(bytes)}$ 个字节数据  
而一个磁道上有 $\text{count(磁道 bytes)}$ 个字节数据  
一共需要转
$$
\frac{\text{count(bytes)}}{\text{count(磁道 bytes)}}
$$
而转一圈需要花
$$
t = \frac{1}{\text{speed(转速)}}
$$

数据传输一共要花
$$
T = \frac{\text{count(bytes)}}{\text{count(磁道 bytes)}} \times \frac{1}{\text{speed(转速)}}
$$

## SPOOLing 技术

SPOOLing，也就是假脱机技术，真脱机技术请去 **查资料**  
真脱机技术 是依靠额外的硬件实现的，假脱机技术就是用软件模拟出 真脱机技术的效果

我们已经有 DMA 方式去缓和 **CPU 的高速** 与 **I/O 设备的低速**
当这个表述 可以出现在 SPOOLing 技术中，这两个东西可以相互替代吗？  
如果你指的，单独是 缓和 **CPU** 与 **I/O设备**，那么完全可以相互替代，但是 SPOOLing 基于其软件实现，还可以实现一种 DMA 做不到的技术 -- 虚拟化共享独占设备

但是，这意味着 SPOOLing 服务会托管整个系统的 I/O ，也就是给整个 I/O 加了一层代理

___
来看一个虚拟打印机的例子，我们知道每次只能有一个进程访问打印机，此时打印机是独占设备

在一个大型linux服务器中，有大约100个进程需要打印，他们得一个一个排队  
假如一个程序会随着 打印的结束 而结束，打印完这100份数据，需要这100个进程等待自己的那份打印完后 再结束

```java
public static void main() {
    printMyData();
}
```

如果一个程序占据的时间过长，对其他程序来说是个麻烦，对系统资源也是个负担，更不用说后续还有可能来更多的打印请求  

如果我们引入异步技术

```java
public static void main() {
    new Thread(() -> printMyData()).start();
}
```

很遗憾，这个程序会比这个线程提前结束，那我们再调整一下

```java
public static void main() {
    Thread thread = new Thread(() -> printMyData());
    thread.start();

    thread.join();
}
```

这样还是和阻塞没区别

这个时候，需要 SPOOLing 程序接手了

```java
public class Main {
    BlockingQueue<Data> spoolingChannel;

    public static void main() {
        spoolingChannel.put(myData);
    }
}
```

交给 SPOOLing 托管，然后程序就可以滚了
