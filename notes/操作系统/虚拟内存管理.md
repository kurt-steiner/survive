## 介绍

前面我们介绍了内存管理，其中我们也知道了现在底层不用分段了，而是用分页管理，那么这个虚拟内存管理也是对 **基本分页管理**的**扩展**

说到虚拟内存中的 "虚拟" ，这里的虚拟类似于编程中的接口概念  
上层可以用相同的方法来使用同一种功能，但是底层对这些功能的实现不一致而已
___
我们知道，从 CF 这款游戏可能有20多个GB，而我们的内存只有16GB  
如果在以往的基本分页中，需要将整个游戏装载入内存中，这肯定不能运行  
现代操作系统都用 **虚拟内存管理** 了，我们完全可以运行这个游戏，还能边玩游戏边开网页，边和好友对喷，虚拟内存管理 中用了什么手段来实现这个功能呢？

其实也简单，用 **懒加载** 就能解决一部分问题  
试想一下，我们在CF的 **运输船地图** 中问候对方的母亲时，系统只需要加载 运输船的地图就行了，其他的地图压根就不需要加载

但是，如果我们已经占用了10多个G的内存呢?  
这也简单，将一些程序的运行数据保存到文件系统或者SWAP分区中即可，等我们退出了游戏再从文件中恢复，其中把谁弄去**置换**是个问题，需要探讨

其实还有一种懒加载的方法也可以用作代码复用的方法，我们玩了一个小时的游戏，不停的问候对方的家人，在旁人眼中，我们是骂的惊天动地，但实际上，我们只有两个核心问候方式 -- 问候对方的母亲，问候对方的父亲  
这种懒加载方法涉及到 **局部性原理**

1. 时间局部性  
我们问候完对方的母亲后，后面可能还会接着问候对方的母亲
2. 空间局部性  
我们问候完对方的母亲后，后面可能会问候对方的父亲 和 二大爷

上述情况中，**懒加载** 可以有开发人员来实现，**局部性原理** 可以由 cache 来实现  
到了 **置换** 这个东西，是由操作系统内核来实现的，这也是我们重点讨论的地方

## 页表项结构调整

我们提出了 **置换** 和 **局部性原理** 是实现虚拟内存的工具  
我们知道，在基本分页管理中，页表项有

1. 页号，隐含在页表的索引中
2. 物理块号

现在我们使用了虚拟内存管理，并且知道了一部分页是被置换到了外存  
在使用 基本分页的页表进行寻址时，我们不知道这个页表项对于的 **页** 是否被载入了内存，我们需要一个 **状态位** 来记录这个页是否被载入内存  
后续我们需要通过局部性原理，把局部性最低的页置换出去，我们再给页表项加个 **访问字段**，记录这个页的访问次数  
我们已经多次提到了置换，当然要知道这个页被置换到了哪去，我们再加一个 **外存地址** 字段

现在页表项被调整为

1. 页号，隐含在页表索引中
2. 物理块号
3. 状态位，记录是否载入内存
4. 外存地址，记录被置换到的外存地址
5. 访问字段，记录访问次数

但是我们还忽略了一个问题  
我们使用快表来映射内存中的页表，同时内存中的页表也记录了没有载入到内存中的页面，这些页面在外存中，我们需要同步 寄存器 - 内存 - 外存 之间的信息  
由于对 外存 内存，寄存器的访问速度 不是一个量级的，完全可以用一个 **修改位** 避免不必要的访存  
现在，页表项位置被更改为

1. 基本地址
    - 页号
    - 物理块号
2. 状态信息
    - 状态位，记录是否载入内存
3. 置换相关
    - 修改位，是否被修改
    - 外存地址，记录被置换到的外存地址
    - 访问字段，记录访问次数，服务于局部性原理

## 访问页面

在前面的章节我们知道了有快表这个东西，这个东西就像 cache 一样处于内存和CPU之间，其中快表项中有页表项的数据副本，并且快表索引与页表索引 不是相同的，在查找时需要**遍历所有快表项**  
既然 快表/寄存器 是 页表/内存 的映射，两者之间需要同步，我们通常在 快表需要 **置换到内存** 时，进行同步

在我们对一个 [页号，页内偏移] 的逻辑地址 进行访问时

1. 这个页号 我们在 快表中遍历到了相应页表项  
    - 我们用快表中的**页表项副本** 对以进行地址转换，得到对应页表项的地址，对其进行访问  
    - 同时，我们需要修改 **页表项副本** 的访问字段/访问次数  
    既然修改了页表项，需要再 修改 **页表项副本** 的修改位，标记数据为脏数据，置换时需要访问内存
2. 没有命中快表，只能去内存中的页表中找对于的页表项
    - 如果页表项对应的页面 **已经被载入内存**  
        1. 我们通过地址变换找到 页面在内存中的地址  
        2. 将这个页面**载入快表**  
        如果快表有空闲，直接载入，如果没有，通过置换算法 置换一个不用的快表项  
        3. 然后，我们在快表中对这个副本进行访问  
        并修改其访问字段和修改位(如果有数据修改)
    - 如果页表项对应的页面 **没有被载入内存**  
        1. 我们发出 **缺页中断**，并请求操作系统调入页面到内存中  
        2. 载入内存中后  
        可能分配的内存不够用，通过某种置换策略对其他页面进行页面置换  
        3. 修改 **内存中的页表** 对应页表项的 访问字段 和 状态位 (可能先要将页表项载入寄存器)
        4. 既然是中断，下次再访问页面时，就会匹配 **命中内存中的页表，并且页面已经被载入内存中** 这种情况

## 页面置换

我们现在才讨论 页面置换，真实姗姗来迟  
我们只有在没有空闲的页框时，才会启动置换算法，这点要谨记

书上介绍的置换方法有5种  
首先是 **最佳置换**，将全局时间未使用的页面换出，但问题是，我们身在居中，无法知晓全局时间中的页面使用情况，现实中无法实现这个算法

再是 一种简单粗暴的 FIFO 方法，即队列，先进先出方法，先载入的页面先被换出，由于局部性原理，很有可能刚被换出的页面 又需要进行访问，造成了多次不必要的缺页中断，我们称之为 Belady 异常

___
**LRU**，最近最久未使用算法，不要看他的名字那么唬人，我们简单讲讲他的实现  
我们为每个页框设置一个 计数器，

```rust
struct Page;

struct PageFrame {
    page_index: usize,
    page: Page,
    count: u8
}

struct Progress {
    page_frames: Vec<PageFrame>
}
```

在每次页面被访问时，计数器置0，没有被访问时，计数器加1

```rust
impl Progress {
    fn access_page(&mut self, page_index: usize) {
        self.page_frames.iter_mut().for_each(|page_frame| {
            page_frame.count += 1;
        });

        let matched_page_frame = self.page_frames.iter_mut()
            .find(|page_frame| page_frame.page_index == page_index);

        if let Some(page_frame) = matched_page_frame {
            page_frame.count = 0;
        }
            
    }
}
```

当需要将页面置换时，会遍历所有页框，将计数器最大的 换出

```rust
impl Progres {
    fn page_frame_index_to_swapout(&self) -> usize {
        self.page_frames.iter()
            .map(|page_frame| {
                page_frame.count
            })
            .enumerate()
            .reduce(|left, right| {
                std::cmp::max_by_key(left, right, |tuple| {
                    tuple.1
                })
            })
            .map(|tuple| {
                tuple.0
            })
            .unwrap_or(0) // 默认第一个页框置换
    }
}
```

在**实际实现** 中，我们有两种实现方法，一是硬件的，而是软件的  
**硬件实现中**， 我们用一组寄存器来实现计数功能，在寸土寸金的CPU内部，这样做的开销太大  
更不要说，有多少个页框，就要多少个计数器了，一些进程的页框可以有几千个
在软件实现中，完全可以把这个计数字段放到 描述页框的结构体中  
我们在每次访问页面时，都有一次额外的内存访问，去修改页表项信息，开销同样很大，相当于性能被拖慢了一倍

___
好在，我们找到了近似与LRU性能的一种方式 -- **时钟置换算法**  
我们在描述页框的结构体中，不用访问字段 记录数字，而是用一个比特位作为访问位，其初始值为 是否已经载入了内存

现在我们这样描述页框和进程

```rust
struct PageFrame {
    page_index: usize,
    page: Page,
    access: bool // 这里就是访问位
}

struct Progress {
    page_frames: Vec<PageFrame>,
    cursor: usize
}
```

每次访问页面时，我们都会将 页框对应的访问位置为 1

```rust
impl Progress {
    fn access_page(&mut self, page_index: usize) {
        let matched_page_frame = self.page_frames.iter_mut().find(|page_frame| page_frame.page_index == page_index);

        if let Some(page_frame) = matched_page_frame {
            page_frame.access = true;
        }
    }
}
```

当我们要 **置换页面**时，我们使用 `cursor` 循环扫描所有页框

1. 如果 `cursor` 指向的页框，其访问位为0，那么
    - 退出循环，将这页框换出
    - `cursor` 往前移动一位
2. 如果 `cursor` 指向的页框，其访问位为1，那么
    - 将页框 访问位 置0
    - `cursor` 往前移动一位
    - 继续扫描

```rust
impl Progress {
    fn page_frame_index_to_swapout(&mut self) -> usize {
        loop {
            let page_frame = &mut self.page_frames[self.cursor];
            if !page_frame.access { // 访问位为0,置换
                let current_cursor = self.cursor;
                self.cursor_next(); // 跳到下一页，方便下次循环继续
                return current_cursor;
            }

            page_frame.access = false; // 将访问位置为 0
            self.cursor_next();
        }
    }

    fn cursor_next(&mut self) {
        if self.cursor == PAGE_FRAME_COUNT {
            self.cursor = 0;
        } else {
            self.cursor += 1;
        }
    }
}
```

我们对比一下 LRU 和 CLOCK  
LRU 中的 **访问字段** 是一个数字，可能是 `u8, u16`，在寸土寸金的CPU内部 成本颇大  
而在 CLOCK 中，**访问字段** 是一个比特位，无论是用硬件实现，还是软件实现，成本都远低于LRU  
并且在置换时，LRU需要找出计数最大的一个页框，这个过程需要遍历进程的所有页框  
而 CLOCK 中，只需要找到一个 访问位 为0的页框 用作置换即可
___
其实，我们还可以对 Clock 算法进行改良  
我们知道，页面被换出时，会和外存中的原始页面进行 数据同步，这个过程由页表项的 **修改位** 决定  
当修改位为1时，我们需要访问外存，这个开销很大  
当修改位为0时，不需要访问外存  
现在我们这样描述页框结构

```rust
struct PageFrame {
    page_index: usize,
    page: Page,
    access: bool,
    modified: bool
}
```

还是循环扫描所有页框

```rust
imp Progress {
    fn page_frame_index_to_swapout(&mut self) -> usize {
        loop {
            let page_frame = &mut self.page_frames[self.cursor];
            if !page_frame.access && !page_frame.modified {
                let current_cursor = self.cursor;
                self.cursor_next();
                return current_cursor;
            }

            if !page_frame.access && page_frame.modified {
                let current_cursor = self.cursor;
                self.cursor_next();
                return current_cursor;
            }

            page_frame.access = false; // 将访问位置为 0
            self.cursor_next();
        }
    }
}
```

在循环中，我们优先置换 未访问过的页面，此时页面有两种情况

1. 访问位 0, 修改位 0，这是最佳置换页面
2. 访问位 0, 修改位 1，这需要进行外存访问，不是很好的置换页面

在尝试匹配后，我们把 页框的访问位 置0，这样在下一次循环中，这样就可以再次匹配 上述两种情况

可以看到，改进后的CLOCK算法虽然有多次内存访存，但省下了大多的外存访问，这个代价值得接收

**补充**  
我们只有在需要置换时才启动置换算法，在判断是否需要置换前，需要看看页框中是否已经加载了需要访问的页面

## 性能考量

### 尽量避免抖动

我们在 FIFO置换 时提到了一嘴 **抖动** ，我们做个极端假设  
如果给进程分配的 页框数量为 1个页面，那每次访问页面时都会产生 缺页中断  
如果给进程分配 无限数量的页框，肯定不会有缺页中断，但是硬件开销实在太大  
我们需要给进程一个合适的页框数量，从而最大程度兼顾 性能和成本  
而为进程分配的 **页框数量**，我们称之为 **驻留集**
___
在描述抖动时，还有两个概念

1. 工作窗口
2. 工作集

在一段时间t内，页框大小 就是工作窗口，其中的页面号集合就是工作集

### 页框分配策略

在为进程分配页框时，页框数量不一定是固定的，也有可能是 **可变的**，如同动态数组  
也有一些严格的系统，为进程分配 **固定的** 页框数量，可以有三种固定策略

1. 所有进程平均分配 页框数量
2. 进程的页框数量 由进程需要的内存大小 决定
3. 进程的页框数量 由进程优先级 决定

### 页面置换策略

一个进程有自己的页框，在需要进行页面置换时，我们都是从进程中寻找满足置换条件的页框，进行置换，再将缺失的页面 调入其中即可，这是 **局部的** 置换

其实我们也可以不需要置换，从系统中找到一个 空闲物理块，将缺失的页面 调入其中即可，这是 **全局的** 置换

### 调入策略

#### a. 什么时候调入页面

在书上和习题中，在程序开始运行时，我们都是一页一页的调入，这就是我们学的 **请求调页策略**  
还有一种方法，在程序开始运行前，直接将需要的页面 都提前载入页框中，这就是 **预调页策略**

#### b. 从哪里调入页面

别看书上写的那么唬人，那么不讲人话  
我们联系实际，在一个大型游戏中，产生的数据 由于是临时的，放在文件系统中得不偿失，可以放到 SWAP 分区中  
而占了大部分空间是 dll 或者一些共享库，能占到几百MB或者几个GB，他们都放在文件系统中，也就是你的游戏安装目录中  
在需要用到这些dll时，从文件系统中调入这些 "页面" 即可
