## 连续分配/基本分段管理

你可以在书上看到，基本分段是和基本分页 在一起讲的，这里我打算把 连续分配和基本分段连在一起讲，也可以说，分段管理就是连续分配

### 基本分段的实现

我们知道，在一个程序中，有

1. 代码段
2. 数据段
3. 栈段
4. 堆段

但是长什么样呢？我们看这样一个程序

```nasm
; 最小的段示例
section .text         ; 代码段
    global _start
_start:
    mov rax, 60       ; exit系统调用
    xor rdi, rdi
    syscall

section .data         ; 数据段
    var1 db 10        ; 已初始化数据
    var2 dw 1000

section .rodata       ; 只读数据段（特殊的数据段）
    hello db "hello", 0

section .bss          ; BSS段
    buffer resb 256   ; 未初始化数据

; 注意：没有堆段的静态定义，堆是运行时动态创建的
```

当我们想要访问 某个段内的数据时，我们的地址使用 [段名, 段内偏移] 的方式，整个地址变换过程是怎么样的的？

我们已经知道

1. 一个进程中有一个 **段表**
2. 我们的逻辑地址是 [段名，段内偏移]，其中段号由编译后隐含处理，变为 [段号，段内地址]
3. 一个段表项有 段号，段长，段在内存中首地址

我们先用软件模拟一遍  
对于逻辑地址构成，我们有

```rust
struct Address {
    segment_index: usize,
    offset_in_segment: u32
}
```

对于段表，我们有

```rust
struct SegmentTableItem {
    segment_length: u32,
    segment_address: u32
}

struct SegmentTable {
    items: Vec<SegmentTableItem> // 段号 由 索引隐含给出
}
```

其中，段表项中的段号，由段表的索引 隐含给出  
一个进程中有一个段表

```rust
struct Progress {
    segment_table: SegmentTable
}
```

在进程中对地址 [段号，段内偏移] 进行地址映射时，我们需要

1. 判断 **段号** 是否超出段表范围，如果 段号 比 段表 长度大，发出越界中断
2. 找到 **段号** 对应的段表项，其中有 **段长** 和 **段的首地址**
3. 判断 **段内偏移** 与 **段长**，如果 段内偏移 比 段长大，发出越界中断
4. 取出 段的首地址，加上 段内偏移 ，得到实际地址

```rust
impl Progress {
    fn find(&self, address: Address) -> Option<u32> {
        if address.segment_index >= self.segment_table.items.len() {
            return None; // 越界中断
        }

        let item = &self.segment_table.items[address.segment_index];
        let segment_length = item.segment_length;
        let segment_address = item.segment_address;

        if address.offset_in_segment >= segment_length {
            return None; // 越界中断
        }

        let raw_address = segment_address + address.offset_in_segment;
        return Some(raw_address);
    }
}
```

好的，现在我们把软件推广到硬件，我们看第2步，有代码

```rust
self.segment_table.items[address.segment_index]
```

在硬件中，找到这个段表项，需要

1. 找到 **段表** 首地址
2. 知道 **段表项** 的长度

现在，对逻辑地址 [段号，段内地址] 进行变换，步骤变为了

1. 判断 **段号** 是否超出段表范围，如果 段号 比 段表 长度大，发出越界中断
2. 找到 **段号** 对应的段表项，段表项地址为 **段表首地址 + 段号 $\times$ 段表项长度**  
其中有 **段长** 和 **段的首地址**
3. 判断 **段内偏移** 与 **段长**，如果 段内偏移 比 段长大，发出越界中断
4. 取出 段的首地址，加上 段内偏移 ，得到实际地址

### 分段的内存分配策略

在内存中，我们装载一个程序，其内存占用为 10KB  
![image](../images/memory-1.png)

这样没有问题，但是其他程序一多，就会变成  
![image](../images/memory-2.png)

总计的空闲空间为10KB，但是这个程序就是装载不进去

我们能对所有程序进行内存调整吗？  
恐怕不能，有些程序在装入内存时，不是在运行时计算其中的逻辑地址，而是在装入时运行其中的逻辑地址，一移动内存，整个程序中的地址就得跟着改，需要花费大量时间，更不要说，在后续还有置换内存与SWAP的场景了

这种情况只能在分配时就根据某种策略进行 内存分配，工程师们针对这种连续内存分配的情况想出了如下策略

1. 首次适应，空闲分区以地址递增的次序排列，查找的时候以第一个匹配分区为主
2. 邻近适应，同上，不过查找的时候，从上一个查找位置继续查找
3. 最佳适应，空闲分区以容量递增的次序排列，查找的时候以第一个匹配的分区为主
4. 最坏适应，空闲分区以容量递减的次序排列，查找的时候以第一个匹配的分区为主

有了这些策略还不够，还需要一个**空闲分区表** 来记录有哪些空闲分区，表项有

1. 地址 `address`
2. 大小 `capacity`

在每次分配内存后，需要调整 空闲分区表  
现在我们用sql的查询语法来描述分配策略

1. 首次适应 `order by address`
2. 邻近适应 `order by address`，和上述相同，不过是在查找时，顺着上一次的位置接着查找
3. 最佳适应 `order by capacity`
4. 最坏适应 `order by capacity desc`

## 基本分页管理

### 地址变换

我们有一台 32位的机器，地址以字节为单位，且在分页管理中，**页面大小与 内存块/行大小相同**，为 4KB  
在这种情况下，能表示 $2^{32}$ 个内存地址，可以表示 $2^{32} / 4KB = 2^{20}$ 个页面，因此我们用 20 位二进制数表示页号  
而一页的大小为 $2^{12}B$ ，因此，我们用 12 位二进制数表示页内偏移

```rust
const PAGE_SIZE: u32 = 4096; //页面大小，4096kb
```

在将 **逻辑地址** 转换为 **物理地址的时候** ，需要将 逻辑地址 转换成 页号 和 页内偏移，我们用这个结构来表示

```rust
struct Address {
    offset_page: usize, // 页号
    offset_in_page: u32 // 页内偏移
}

impl Address {
    fn from(logic_address: u32) -> Self {
        let offset_page = logic_address / PAGE_SIZE;
        let offset_in_page = logic_address % PAGE_SIZE;

        Self {
            offset_page: offset_page as usize,
            offset_in_page
        }
    }
}
```

我们将 逻辑地址 转换为 `Address` 结构，其中用到

1. 整除 页面大小 获取 页号(0-based)
2. 取模 页面大小 获取 页内偏移(0-based)

那又如何使用这个结构获取 物理地址呢？  
我们知道，一个进程中有他自己的页表，这个页表可以看作一个数组，其中的元素存储了物理块号

```rust
struct PageTable {
    length: u32, // 页表长度
    data: [u32; PAGE_TABLE_SIZE] // 页表数据，对应 map<页号,内存块号>
}

struct Progress {
    page_table: PageTable
}
```

在这个进程中使用逻辑地址 寻址时

```rust
impl Progress {
    fn find(&self, logic_address: u32) -> Option<u32> {
        let address = Address::from(logic_address);
        if address.offset_page >= self.page_table.length as usize {
            return None;
        }

        let offset_page = address.offset_page;
        let offset_memory_block = self.page_table.data[offset_page]; // 内存块号
        let address_memory_block = offset_memory_block * PAGE_SIZE; // 内存块号地址
        let raw_address = address_memory_block + address.offset_in_page;
        return Some(raw_address);
    }
}
```

我们需要

1. 转换逻辑地址 为 `Address` ，其中带有 页号 和 页内偏移
2. 判断 页号 是否超出了 索引范围，如果超出，发出越界中断
3. 从 进程的页表 中取出对应 **页号/索引** 的数据，这个数据存储的是内存块号
4. 内存块号 x 内存块大小 获取 内存块号的地址，由于本例中 内存块大小和页面大小 相同，内存块号地址 可以看作 内存块号 x 页面大小
5. 对内存块号地址 加上 页内偏移，得到真实物理地址

上述我们是用软件的视角去看待 地址变换 的过程，在硬件实现中略有不同，尤其是对 页表的访问  
在软件视角下，我们已经知道了 **页表的长度** 用作判断 越界访问，但在硬件视角下，我们还需要知道 页表对象 所在的 **内存首地址**

我们再次整理整个过程，对一个逻辑地址 `logic_address` 进行寻址，硬件会

1. 计算页号 `offset_page = logic_address / PAGE_SIZE`
2. 计算页内偏移 `offset_in_page = logic_address % PAGE_SIZE`
3. 从进程页表中获取存储的**内存块号**
    - 判断是否发生缺页中断  
    `offset_page >= length(page_table)`
    - 计算页表项地址  
    `address_page_entry = address(page_table) + offset_page * PAGE_SIZE`
    - 获取页表项内容，即内存块号  
    `offset_memory_block = *address_page_entry`
4. 获取内存块号 对应的物理地址  
    `address_memory_block = offset_memory_block * MEMORY_BLOCK_SIZE`  
    或者  
    `address_memory_block = offset_memory_block * PAGE_SZE`
5. 对内存块号地址 加上 页内偏移，获取真实地址  
    `address = address_memory_block + offset_in_page`

## 补充: 物理上的分段已经被废弃，逻辑上的分段还在使用

我们知道，分段和分页 针对的是内存的分配，两者之间不能同时使用，只能通过段页式方式进行结合

我们还知道，linux系统是用的分页管理方式，但他可以写分段的汇编程序，就像上述代码

```nasm
; 最小的段示例
section .text         ; 代码段
    global _start
_start:
    mov rax, 60       ; exit系统调用
    xor rdi, rdi
    syscall

section .data         ; 数据段
    var1 db 10        ; 已初始化数据
    var2 dw 1000

section .rodata       ; 只读数据段（特殊的数据段）
    hello db "hello", 0

section .bss          ; BSS段
    buffer resb 256   ; 未初始化数据

; 注意：没有堆段的静态定义，堆是运行时动态创建的
```

这就很奇怪了，这让人认为linux在底层同时使用了分段和分页，其实不是的  
简单来说，intelx86系列芯片强制要求分段，linux开发者们，用了某种黑魔法，把分段检测绕过去了，最终实现的是分页管理  
现在的分段变为了逻辑视角/用户视角，各种现代操作系统不在内存分配上使用分段管理

## 补充: 程序的链接方式 和 装入方式

程序，由程序员编写在源代码里，通过编译，链接后，变成了某种可执行文件(elf, exe 或者 二进制文件)，在需要运行时，会先把可执行文件装入内存  
现在，从源代码到一个任务，需要经过

1. 编译
2. 链接
3. 装入内存
4. 运行

如果我们把一个任务看作一份拼图，那么编译就是把源代码 变成拼图的块，链接就是把各个块拼凑成一部分拼图，而链接有这么几种

1. 静态链接  
指在装入内存前提前把所有块 拼成完整的拼图  
这样做可以隔离 共享库的依赖地狱，详见在ubuntu上装软件的时候，一个不小心把桌面卸载了  
不过这样做，会使链接后的可执行文件大小相对较大
2. 动态链接  
上述我们说到，静态链接 会使链接后的可执行文件大小相对较大，一个比较明显的例子是，在linux的开源软件上，可执行文件的大小比windows上的小了不止一点，比如在 windows上几个GB的文件，在linux上可能就只需要 10几MB  
动态链接就是在你需要的时候再拼出一份拼图来，不过什么时候拼也有说法
    - 在linux系统中，一般选择在装入内存时链接，其共享文件以 so 结尾
    - 在macos系统中，一般选择在运行时链接，其共享文件以 dylib 结尾，由于macos强悍的性能和特制的apfs文件系统，这种方式对程序的性能没有多少影响
    - 还有一种在运行时，手动链接外部共享库的情况，在 linux/macos 中都有，那就是 `dlopen` 函数

现在链接完毕，该把可执行文件装入内存中了  
我们知道，在编译时，其中的地址都是逻辑地址，在真正运行时需要的是物理地址，我们需要在运行前 就将这些地址转换完毕

1. 在早期单道程序中  
我们压根就不用担心这个东西，因为只有一个程序会运行，直接将地址写死也没问题
2. 但在多道程序中  
由于有不同程序在运行，其相同的逻辑地址不能映射到相同的物理地址，不然会造成不可预测的错误
    - 一种很容易想到的方法，是在装入内存时 就开始对地址进行转换，这肯定要花费一定时间
    - 还有一种懒加载的方法，装入内存就装入内存，让地址转换发生在运行时

后续的内存置换中，我们会发现将页面换回来时，不一定会放到相同的内存块中，也就是说，物理地址变了，这个时候不能在 装入内存时对地址进行转换，而是需要在运行时 重新算一遍

## 补充: 快表

我们在学组成原理的时候，就知道了这个 cache 是对内存的映射，是对内存的快速访问  
我们回顾上述查询页表/段表的过程，发现 页表/段表 都是在内存中存储的，会多次访问内存  
既然是多次访问内存，那么我们自然而然就想到了在 靠近CPU的寄存器 中映射一部分 页表/段表  作为快表
由于快表映射的表索引可能不连续，需要遍历查找，没有找到时，再去内存中去找  
其中快表项有

1. 页号/段号
2. 页的备份/段的备份
3. 状态信息，比如是否有效，是否脏数据
