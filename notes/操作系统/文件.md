## 接口与接口实现

在C语言中进行文件操作时，我们都是认为 文件是连续存储的，但这只是文件接口给我们的错觉，在物理层面不一定是连续存储的  

文件的接口需要我们实现

1. 连续存储
2. 文件读写
3. 文件保护

我们需要重点探讨 连续存储 的逻辑实现

### 1. 列表方式

文件的底层实现，其实就是数据结构的一种应用  
我们要存储一组数据，可以是用数组存储，在需要添加数据 或者 删除数据时，数组做不到，动态数组却可以，不过需要 `realloc`，重新分配更大/更小的存储块，开销和成本极大，实际上不会用这种方式

列表还有一种 链表的形式，其节点结构可以这样表示

```c
struct ListNode {
    uint32_t data;
    struct ListNode * next;
}
```

如果用这种形式去存储文件，一个存储块中除了要记录数据外，还要记录下一个块的地址，存储效率不高  
有种改进方式，我们将一组存储块作为节点，在最后一个节点处存储下一组的首地址即可

### 2. 字典方式/索引方式

内存管理中我们已经用分页管理替代了连续分配  
类似的，在文件中，我们也将磁盘分为一块一块，用某一块作为索引，记录文件的所有存储块  
这个索引块就是一个数组，其中记录了 各数据块的索引  
现在存储块有两种用途，一是作为索引块，而是作为数据块，我们描述一下

```rust
const BLOCK_SIZE: usize = 4096; // 一个物理存储块的大小是4KB
type RawBlock = [u8; BLOCK_SIZE];



pub(crate) struct DataBlock(pub RawBlock); // 用来存储数据，每个可以存储 4KB 个数据，结构体很大，不要作死运行！！
pub(crate) struct IndexBlock(pub Vec<Address>); // 用来存储索引，每个最多可以存储 1024 个，即 [u32; 1024]，不一定会存满，所以反序列化时 用 Vec
```

这个索引块虽然是个数组，但他的功能完全可以用 `map<Index, Address>` 来描述

#### a. 索引如何工作

索引，有一级的，有二级的，我们来看看如何使用这些索引查找到文件的所有存储块

我们先描述下操作系统

```rust
pub(crate) struct OperatingSystem {
    pub(crate) blocks: Vec<RawBlock>, // 由于硬盘容量不一定，这里用 动态/不定长 数组表示
    pub(crate) stream: Sender<DataBlock>,
}
```

在描述下文件的控制块，即文件的元信息

```rust
#[derive(Clone, Copy)]
pub(crate) struct FileControlBlock {
    pub(crate) size: u64,
    pub(crate) is_directory: bool,
    pub(crate) is_file: bool,
    pub(crate) is_block: bool,
    pub(crate) is_symlink: bool,
    pub(crate) is_hidden: bool,

    pub(crate) index_start_block: usize
}
```

其中，每个FCB都存储了其索引块的索引/地址

操作系统使用一级索引查找文件存储块时，

```rust
impl OperatingSystem {
    fn transfer_in_one_level(&self, index_start_block: usize) {
        let block_raw = self.blocks[index_start_block];
        let block_index = block_raw.as_index();
        block_index.0.iter().for_each(|index| {
            let index_block_data = *index as usize;
            let block_raw_inside = self.blocks[index_block_data];
            let block_data = block_raw_inside.as_data();
            self.stream.send(block_data).expect("fuck");
        });
    }
}
```

而二级索引，就是索引的索引，本来是一次间接查找，现在两次间接查找

```rust
impl OperatingSystem {
    fn transfer_in_two_level(&self, index_start_block: usize) {
        let block_raw = self.blocks[index_start_block];
        let block_index_one_level = block_raw.as_index();

        for index_one_level in block_index_one_level.0.iter() {
            let index_one_level = *index_one_level as usize;
            let block_two_level_raw = self.blocks[index_one_level];
            let block_two_level = block_two_level_raw.as_index();

            for index_two_level in block_two_level.0.iter() {
                let index_block_data = *index_two_level as usize;
                let block_two_level_raw = self.blocks[index_block_data];
                let block_data = block_two_level_raw.as_data();
                self.stream.send(block_data).expect("fuck");
            }
        }
    }
}
```

![image](../images/hybrid-index.png)

教材中用这么一张图表示混合索引，其实这不是索引块的结构，而是FCB的结构，现在我们调整一下FCB被反序列化后的结构

```rust
impl OperatingSystem {
    fn transfer_in_hybrid(&self, fcb: FileControlBlock) {
        assert!(fcb.is_file); // 确保不是目录

        for index in fcb.indexs_directly_access_block.iter() {
            let index = *index as usize;
            let block_raw = self.blocks[index];
            let data_block = block_raw.as_data();
            self.stream.send(data_block).expect("fuck");
        }

        if let Some(index_one_level) = fcb.index_one_level {
            self.transfer_in_one_level(index_one_level as usize);
        }
        
        if let Some(index_two_level) = fcb.index_two_level {
            self.transfer_in_two_level(index_two_level as usize);
        }

        if let Some(index_three_level) = fcb.index_three_level {
            // 懒得写，好麻烦
        }
    }
}
```

我们声明一个反序列后的结构体表示 这种列表元素

```rust
    pub(crate) fcb: FileControlBlock
}
```

再表示下 可以从原始块解析出的 "目录块"

```rust
pub(crate) struct DirectoryBlock(pub Vec<DirectoryEntry>);  // 用来存储目录表，数量看 DirectoryEntry 的大小
```

在当前目录查找文件时，给出文件名

```rust
impl OperatingSystem {
    fn lookup_in_directory(&self, filename: String, directory_fcb: FileControlBlock) -> Option<FileControlBlock> {
        let block_raw = self.blocks[directory_fcb.index_start_block];
        let block_directory = block_raw.as_directory();
        block_directory.0.iter().find(|entry| entry.filename == filename)
            .map(|entry| entry.fcb)
    }
}
```

接下来通过 FCB 进行操作就好

**补充**  
注意到在列表项中，我们存储的是 FCB 的指针，而不是整个FCB，还是那句话 --- **寸土寸金** ，这样可以让一个目录多放点文件  
其实还有个原因，在文件共享中，有个硬链接方式，到时候就不是存储 `Ptr<FCB>` 了，而是 `Ptr<IndexBlock>/Ptr<IndexNode>`  
这个时候存整个 FCB 结构，内存对齐是个大问题，还不如直接来个64位数字作为地址来得实在

## 文件共享

在 unix 多用户环境下，如果在室友的目录下有个 **邻りの女子大生 恶魔の肉调教-4k.avi** ，并且其占用存储为 10GB，现在我也想看怎么办  
如果我直接把这个文件复制到我的目录里，虽然也是个方法，但是整个系统会有不必要的10GB拷贝，在共享小文件时，这种方法还行，大文件就没那个必要了  
那我用类似windows上的快捷方式呢？我们可以这样操作

```bash
ln -s "/home/other/邻りの女子大生 恶魔の肉调教-4k.avi" "/home/me/学习资料.avi"
```

其中 `-s` 参数表示这是一个软链接  
既然有软链接，那就有硬链接，这两个都是文件共享的方式，有什么区别吗

如果我们使用软链接 链接到这个 avi，在我的用户目录的存储块中，有

|filename| Ptr |
|:--:|:--:|
| 学习资料.avi | `Ptr<FCB>`|

系统为这个链接文件 也创建了一个 FCB ，其中 `FCB.is_symlink = true`, FCB 的内容中存储了 **被链接文件的地址**  
需要访问时，系统需要再走一趟才能访问到真正的文件  
如果 **被链接的文件** 被删除了，此时指针变成了**悬空指针**  
再次访问链接文件时，系统因为找不到文件位置而告知用户，文件不存在

如果我们使用硬链接，在用户目录存储块中，有

|filename| Ptr|
|:--:|:--:|
| 学习资料.avi | `Ptr<IndexBlock>`|

当需要访问时，由于存储的是索引块的地址，速度会比软链接快许多  
如果 **被链接的文件** 被删除了，这只能删除指针，而没有删除所有块  
假如删除了这个文件的所有硬链接，那才是把文件真正删除了，我们需要给 **索引块** 一个引用计数 字段去实现这个真正删除

但是这种硬链接方式有种缺陷 -- 不能链接目录  
因为目录的判定需要 FCB 来判断，而指针指向的是 索引块，因此不能判断是否是 目录

## 空闲分区管理

前面我们在学习内存管理的时候，我们都只关注了 如何进行 地址转换 和 页面置换  
却忘了一件非常重要的事 -- 巧妇难为无米之炊  
再好的方法，没有足够的空闲内存也是白搭  
这个被我们忽略掉的东西就是 -- **空闲内存管理**，教材上因为没有讲这个东西，所以先放到一边去  
其实在文件系统中，也有一套 **空闲分区管理** ，我们需要对其重点探讨

### 成组链接法

![image](../images/group-index.png)

王道的书上没有讲多少，这里我讲下实现  
系统将一个存储块作为超级块 载入内存

![image](../images/group-index-2.png)

在每一个存储块中，都有一个空闲块的计数，还有固定大小的空闲栈，记录了空闲块在存储中的索引  
超级块 空闲栈的第一个元素中，即 `S.free[0]`，放置的是索引块  
其他地方放其他空闲块的索引

```rust
pub(crate) struct GroupIndexBlock {
    count: usize,
    free_stack: [Address; FREE_STACK_SIZE]
}
```

在将空闲块分配给系统时，我们需要及时调整超级块

```rust
impl OperatingSystem {
    // 分配空闲块
    fn allocate_free_block(&mut self) -> Address {
        if self.super_block.count() == 1 {
            let poped = self.super_block.pop();
            let next_block_raw = self.blocks[poped as usize];
            let next_block = GroupIndexBlock::from(next_block_raw);
            self.super_block = next_block;
            return self.super_block.pop();
        } else {
            return self.super_block.pop();
        }
    }
}
```

当超级块的空闲块数量 只有一个 时，即当前超级块已经没有空闲块了，当他指向了一个索引块
现在我们从这个索引块中拿取空闲块，并将这个索引块 作为 超级块

这段代码还有一种形式

```rust
impl OperatingSystem {
    fn allocate_free_block(&mut self) -> Address {
        // 对上面代码的改进
        let poped = self.super_block.pop();
        let mut result = poped;
        if self.super_block.is_empty() {
            let next_block_raw = self.blocks[poped as usize];
            let next_block = GroupIndexBlock::from(next_block_raw);
            self.super_block = SuperBlock {
                block: next_block,
                block_index: poped as usize
            };
            
            result = self.super_block.pop();
        }

        return result;
    }
}
```

系统用完了某个存储块，要对其进行回收，如果超级块还有位置的话，直接压入即可  
如果超级块已满，需要将这回收块作为超级块，并将上个超级块的索引压入其中

```rust
impl OperatingSystem {
    fn recycle_free_block(&mut self, block_index: usize) {
        if self.super_block.is_full() {
            let recycled_block_raw = self.blocks[block_index];
            let mut recycled_block = GroupIndexBlock::from(recycled_block_raw);
            recycled_block.clear();
            let original_super_block_index = self.super_block.block_index;
            recycled_block.push(original_super_block_index as u32);

            self.super_block = SuperBlock {
                block: recycled_block,
                block_index
            };
        } else {
            self.super_block.push(block_index as u32)
        }
    }
}
```

由于我们的实现中，无法直接找到 超级块的索引，只好调整下超级块和操作系统的结构体

```rust
pub(crate) struct SuperBlock {
    block: GroupIndexBlock,
    block_index: usize
}
```

```rust
pub(crate) struct OperatingSystem {
    pub(crate) blocks: Vec<RawBlock>, // 由于硬盘容量不一定，这里用 动态/不定长 数组表示
    pub(crate) stream: Sender<DataBlock>,
    pub(crate) super_block: SuperBlock
}
```
