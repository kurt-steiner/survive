## 部分匹配表

整个KMP算法中，从始至终都没有对 **主串** 做过一点东西，都是对 **子串** 做的

对于子串 **abcac** ，我们这样生成部分匹配表

|字符串|前缀|后缀|公共相等前后缀长度|
|:--:|:--:|:--:|:--:|
|a | null| null | 0 |
|ab | a | b | 0 |
|abc | a,ab |c,bc | 0 |
|abca| a,ab,abc | bca, ca, a |1 |
|abcac| a, ab, abc, abca | bcac, cac, ac, c| 0 |

于是有 部分匹配表

|编号| 1 | 2 | 3 | 4| 5|
|:--:|:--:|:--:|:--:|:--:|:--:|
|char |a |b |c|a |c |
|pm | 0 | 0 | 0 | 1| 0 |

在匹配主串时，移动的位数是 **已匹配长度 - 对应匹配表的值**  
比如在这种情况下

|主串| a |b | a| b| c|a |b |c |a | c| b | a |b |
|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|
| 子串|a |b |c | a| c| | | | | |  |  | |

已经匹配了 *ab*，其长度为2, *ab* 对应部分匹配值为 0，将子串向前移动 `2-0=2`位，得到

|主串| a |b | a| b| c|a |b |c |a | c| b | a |b |
|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|:--:|
| 子串| | |a |b |c | a| c| | | |  |  | |

重复上述过程即可

## 部分匹配表的调整

上述过程中，我们知道
> 在匹配主串时，移动的位数是 **已匹配长度 - 对应匹配表的值**

对于一个 1-based 的子串串匹配过程，用代码表达就是

```julia
movement = index - 1 - pm[index - 1]
```

___
大家知道为什么 **C语言** 中为什么数组是从0开始的吗?  
本来大家都是在语言层面，默认从1开始，其实在底层，也是从0开始的  
如果你从1开始，索引数组的时候

```c
array[index]
```

底层就要操作一遍

```c
array[index - 1]
```

可以看到，多了个减法，那个时候的硬件性能不是很好，一次计算就要花费一次可观的时间  
C语言的发明者为了极限压榨性能，把这个减法省掉了，于是就有了数组从0开始的惯例
___

回到正题，同样的，为了减少不必要的运算，完全可以把 部分匹配表的值 向前移动，上述的表变成

|编号| 1 | 2 | 3 | 4| 5|
|:--:|:--:|:--:|:--:|:--:|:--:|
|char |a |b |c|a |c |
|pm | -1 | 0 | 0 | 0| 1 |

于是子串的移动变为了

```julia
movement = index - 1 - pm[index]
```

其中，空位我们用 -1 来填充  
可以看到，我们 **减少了一次** 减法运算  
其实，上述移动的代码也可以是

```julia
movement = index - (pm[index] + 1)
```

此时，我们需要将部分匹配表的值都 `+1` ，变为

|编号| 1 | 2 | 3 | 4| 5|
|:--:|:--:|:--:|:--:|:--:|:--:|
|char |a |b |c|a |c |
|pm | 0 | 1 | 1 | 1| 2 |

这样匹配时，结果不会变
